8. Critical Review
8.1 Achievement of Objectives
All eight project objectives were met. Functional and non-functional requirements were defined and tracked through Table 4.1, and the implemented system captures live packets, performs real-time visualisation, and provides anomaly detection across port scanning, high PPS, and suspicious port activity. A clear, grid-based dashboard interface was designed with configurable dark and light themes, and the system was tested across unit, functional, and performance dimensions.
However, meeting objectives on paper does not necessarily equate to meeting them to a high standard in every respect. Anomaly detection thresholds, for example, were set as fixed values and require manual tuning for different network environments — this means that while the feature was implemented, its real-world reliability is limited without further calibration. Similarly, the tool focuses primarily on IPv4 traffic, which means a significant portion of modern network communication — IPv6 accounts for a growing share of internet traffic globally — falls outside the scope of what the system can analyse. These are not failures of implementation but they do represent gaps between what was delivered and what a production-ready tool would require, and it is important to acknowledge that distinction honestly.
The Wireshark comparison conducted in Section 5 provides the most useful evidence base for this evaluation. The Network Traffic Visualiser captures broadly equivalent data fields at the Ethernet, IPv4, TCP, UDP, and ICMP layers, which demonstrates genuine technical achievement. However, Wireshark also provides additional fields — such as Differentiated Services, ECN flags, TCP Window Size, and nanosecond-precision timestamps — that the visualiser does not currently surface. For an educational tool aimed at beginners, these omissions are justifiable; for a professional monitoring tool, they would be significant limitations. Understanding that distinction is itself a mark of maturity in evaluating the project honestly.

8.2 Strengths of the System
The most technically significant strength of this project is the decision to implement packet capture using Python's native socket library with AF_PACKET and SOCK_RAW sockets, combined with AsyncSniffer for non-blocking capture, rather than relying on a high-level abstraction library. This approach delivers better performance under load and demonstrates a genuine understanding of how network data moves through the operating system. This is supported in the technical literature — raw socket implementations consistently outperform high-level sniffing libraries in CPU efficiency during live capture, which is critical for a real-time visualisation tool where capture and display must occur concurrently without one blocking the other.
The professional, polished UI with configurable dark and light themes is also a notable strength, particularly given that the primary goal of the project was accessibility and clarity for non-expert users. Colour selection was grounded in pre-attentive processing theory, as documented in Section 2.3, ensuring that protocol types are visually distinguishable before conscious analysis is required. This is not a superficial aesthetic choice — it reflects an understanding of how human perception interacts with data displays, which Healey and Enns (2012) identify as a foundational principle of effective visualisation design.
The demo mode, which enables classroom use without administrator privileges, also demonstrates thoughtful consideration of the tool's real-world deployment context — something that purely technical projects often overlook entirely.

8.3 Limitations and Challenges
The most significant limitation of the system is that it requires administrator privileges on Windows for live packet capture. This creates a meaningful barrier to deployment in educational environments where students are unlikely to have elevated permissions on institutional machines. While this constraint is a consequence of how operating systems handle raw socket access at the kernel level rather than a design flaw per se, it is a limitation that was known early in development and could have been partially mitigated by investing more time in a web-based interface that offloads capture to a privileged server component. That architectural choice was considered but deprioritised in favour of completing the desktop implementation — in retrospect, this may have been the wrong trade-off given the tool's stated educational purpose.
Performance degradation above 500 PPS on lower-specification hardware is also a genuine limitation that deserves honest reflection. The 1-second refresh interval was selected based on guidance from Few (2013), who recommends update intervals of one to two seconds to balance situational awareness with visual stability. In testing, this interval was validated informally, but no formal performance benchmarking was conducted against different hardware configurations. A more rigorous evaluation — for instance, testing across three different hardware tiers and documenting capture accuracy at varying traffic loads — would have provided stronger evidence that the performance threshold was correctly identified.
No persistent data storage is another limitation worth examining critically. Captured statistics are lost when the application closes, which means the tool cannot be used for post-session review or trend analysis over time. This was an acknowledged design decision but one that significantly limits the tool's utility for the network forensics and capacity planning use cases described in Section 6. Future versions should implement PCAP export as a minimum, which would also enable interoperability with Wireshark for deeper post-capture analysis.

8.4 What Would Be Done Differently
Reflecting on the project as a whole, three things would be approached differently if the project were undertaken again.
First, the literature review would be engaged with more deeply from the beginning rather than treated as background context. The theoretical frameworks that ultimately influenced the visualisation design — Shneiderman (1966), Healey and Enns (2012), Few (2013) — were identified during development but could have shaped design decisions more deliberately had they been reviewed before implementation began. Grounding technical choices in established research from the outset, rather than finding supporting literature retrospectively, produces more coherent and defensible design rationale.
Second, the project management approach would be refined. Kanban was the right choice for a solo developer project — Zayat and Senvar (2020) confirm that Kanban is better suited to continuous-flow environments than Scrum's fixed sprint structure — but the implementation of it could have been more disciplined. Large features such as the web-based interface and IPv6 support were listed as tasks without being broken into sub-tasks, which made it difficult to track their progress accurately and contributed to them slipping out of scope. More granular task decomposition from the start would have made scope risks visible earlier.
Third, user evaluation would be built into the project plan as a formal activity rather than handled informally during testing. Even a structured session with two or three peers from outside the project — asked to interpret the dashboard output without guidance — would have generated valuable evidence about whether the tool genuinely achieves its goal of accessibility for non-expert users. This kind of formative evaluation is standard practice in user-centred design and its absence is a gap in the overall evidence base for the project's success.

8.5 Future Improvements
The most impactful future development would be a web-based interface that eliminates the need for local installation and administrator privileges, directly addressing the primary deployment barrier identified in Section 8.3. This would also enable cross-platform support for macOS and Linux without requiring environment-specific configuration.
IPv6 support and expanded protocol coverage — including ARP, DNS, and TLS — would substantially broaden the tool's analytical capability and bring it closer to parity with professional tools. Machine learning-based anomaly detection, replacing the current fixed-threshold approach, would make the detection engine more adaptive and reduce the manual tuning currently required across different network environments. Historical traffic playback from PCAP files would support post-incident review and enable the tool to be used in structured educational exercises where students analyse pre-recorded traffic rather than live capture.
